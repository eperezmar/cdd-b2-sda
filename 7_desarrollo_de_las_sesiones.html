7. Desarrollo de las sesiones

TAREA 1: Presentación del reto y análisis de requisitos del banco de alimentos

Temporalización: 1 sesión (2 horas)

Contenido propuesto

  • Presentación introductoria (formato Slides/Canva) sobre la realidad de los bancos de alimentos, retos de digitalización y el proyecto que desarrollará el alumnado.
  • Dossier de requisitos funcionales (PDF editable) simulando las necesidades del cliente (registro de usuarios, gestión de productos, seguimiento de donaciones y stock).
  • Infografía base (Canva) con roles de usuario y flujos básicos del sistema (donante ↔ banco ↔ receptor).

Actividades del alumnado

  • Leer y analizar el dossier funcional.
  • Crear una infografía grupal sobre las necesidades clave detectadas (herramientas: Canva o Genially).
  • Realizar un documento resumen con los requisitos funcionales mínimos viables y posibles mejoras.
  • Foro de lluvia de ideas (Teams) sobre dudas y ampliaciones posibles del proyecto.

Material de apoyo

  • Video breve sobre “¿Cómo funcionan los bancos de alimentos?” (YouTube, max. 5 minutos)
  • Plantilla base para análisis de requisitos (Word/Docs)

Diccionario de términos

  • Requisito funcional: Funcionalidad concreta que debe ofrecer un sistema
  • MVP (Producto Mínimo Viable): Versión básica de un producto que cumple su función principal
  • CRUD: Acrónimo para crear, leer, actualizar y eliminar datos en una app

Preguntas de repaso

  • ¿Quiénes son los usuarios del sistema?
  • ¿Qué operaciones necesita gestionar el banco?
  • ¿Qué diferencias hay entre requisitos funcionales y no funcionales?

Soporte digital

  • Microsoft Teams: entrega y debate
  • Office 365: trabajo colaborativo
  • Canva: diseño infográfico

TAREA 2: Diseño de la base de datos y arquitectura MVC del sistema

Temporalización: 2 sesiones (4 horas)

Contenido propuesto

  • Ejemplo de diagrama entidad-relación
  • Tutorial visual (vídeo breve) sobre el patrón MVC en aplicaciones web (YouTube, 6-8 min).
  • Diagrama de estructura de carpetas en un proyecto Java/Spring o Node.

Actividades del alumnado

  1. Diseñar su modelo de datos en Draw.io (usuarios, productos, donaciones…).
  2. Especificar claves primarias/foráneas y relaciones (1:N, N:N).
  3. Crear un borrador de arquitectura MVC del sistema y carpetas del proyecto.
  4. Validar el diseño con docente o compañeros.

Material de apoyo

  • Plantilla de modelo de datos
  • Guía rápida: “MVC en 10 pasos”
  • Vídeo “Cómo crear tu primer diagrama ER en Draw.io”

Diccionario de términos

  • Clave primaria: Identificador único para una fila en una tabla
  • MVC: Patrón de arquitectura: Modelo, Vista y Controlador
  • Diagrama ER: Representación gráfica de entidades y sus relaciones

Preguntas de repaso

  • ¿Qué relación hay entre usuarios y donaciones?
  • ¿Qué elementos se gestionan en la vista, el modelo y el controlador?

Soporte digital

  • Draw.io o Lucidchart: diseño de BD
  • Teams/OneNote: entrega y corrección
  • Trello: organización de tareas del grupo

TAREA 3: Implementación por módulos: usuarios, productos, donaciones, stock

Temporalización: 2 sesiones (4 horas)

Contenido propuesto

  • Código ejemplo básico de un CRUD en Spring Boot / Node
  • Videotutorial (propio o externo) sobre uso de Postman para pruebas API
  • Infografía de buenas prácticas en control de versiones con GitHub

Actividades del alumnado

  1. Crear endpoints CRUD para cada entidad.
  2. Implementar validaciones básicas.
  3. Probar con Postman y grabar un mini tutorial del uso.
  4. Subir el código al repositorio GitHub con commits comentados.

Material de apoyo

  • Plantilla básica para endpoint REST
  • Guía de uso de Postman
  • Ejemplo de repositorio documentado

Diccionario de términos

  • Endpoint: Punto de acceso a una funcionalidad de una API
  • ORM: Herramienta para mapear objetos y tablas de BD
  • Commit: Registro de un cambio realizado en el código en Git

Preguntas de repaso

  • ¿Cómo se prueba que una API funciona correctamente?
  • ¿Por qué es importante usar control de versiones?

Soporte digital

  • IDE local (VS Code, IntelliJ)
  • Postman
  • GitHub para versionado

TAREA 4: Revisión de seguridad, testeo y documentación

Temporalización: 2 sesiones (4 horas)

Contenido propuesto

  • Ejemplo de archivo Swagger documentado
  • Video tutorial sobre uso básico de Swagger/OpenAPI
  • Check-list de seguridad básica

Actividades del alumnado

  1. Añadir autenticación (JWT o sesiones)
  2. Documentar los endpoints con Swagger
  3. Testear funcionalidades y errores comunes
  4. Hacer revisión cruzada con otro grupo

Material de apoyo

  • Plantilla Swagger básica
  • Ejemplo de middleware de autenticación
  • Código base de test en JUnit o similar

Diccionario de términos

  • JWT: Token que permite autenticación sin mantener sesión
  • Test unitario: Prueba que verifica el funcionamiento de una unidad del código
  • Swagger: Herramienta para documentar APIs de forma visual e interactiva

Preguntas de repaso

  • ¿Qué rutas deben protegerse en una API?
  • ¿Qué información mínima debe aparecer en la documentación técnica?

Soporte digital

  • IDE
  • Swagger UI
  • Teams o GitHub para revisión de código

TAREA 5: Presentación y despliegue del proyecto

Temporalización: 1 sesión (2 horas)

Contenido propuesto

  • Video modelo de presentación final de un proyecto web
  • Plantilla para README en GitHub (con badges, instrucciones, etc.)
  • Guía visual para despliegue en Heroku o servidor local

Actividades del alumnado

  1. Subir versión final del proyecto a GitHub
  2. Preparar presentación oral y mostrar app funcionando
  3. Compartir enlace del repositorio + demo

Material de apoyo

  • Checklist de revisión final del proyecto
  • Guía: “Cómo preparar tu README profesional”
  • Diapositiva modelo para presentación final

Diccionario de términos

  • Despliegue: Publicación de una app para su acceso por parte de usuarios
  • README: Documento que acompaña un repositorio y explica cómo usarlo
  • Demo: Demostración del funcionamiento de una app

Preguntas de repaso

  • ¿Qué pasos hay para desplegar tu app en la nube?
  • ¿Qué elementos debe tener una buena presentación final?

Soporte digital

  • GitHub
  • Heroku / Docker / XAMPP
  • PowerPoint o Canva
  • Plataforma del centro para subir vídeo o demo
